# Чеклист развёртывания на сервере

Пошаговая инструкция для развёртывания магазина «Эпатаж» на production-сервере.
Выполнять по порядку. Отмечай галочками по мере выполнения.

---

## ⚠️ Критичные доработки ПЕРЕД боевым запуском

Эти три пункта обязательны — без них на проде будут реальные проблемы.
Подробности в шаге 8, но осознать нужно заранее.

- [ ] **Чтение реального IP за nginx** — иначе защита от перебора паролей
  заблокирует всех пользователей разом.
- [ ] **Том для загруженных фото** — иначе при каждом обновлении кода
  фото товаров будут пропадать.
- [ ] **Резервные копии базы** — настроить сразу, а не «потом».

---

## 0. Что понадобится заранее

- [ ] Арендованный VPS (Ubuntu 22.04+), минимум 2 ГБ RAM
- [ ] Доменное имя, направленное на IP сервера (A-запись)
- [ ] Боевой магазин в YooKassa с реальными ключами (shopId + секретный ключ)
- [ ] Доступ к серверу по SSH

> **Важно про секреты:** все боевые пароли и ключи генерируются заново
> и хранятся только в файле `.env` на сервере. Никогда не коммить их в git
> и не передавать в переписке.

## 1. Подготовка сервера

- [ ] Подключиться по SSH:
  ```bash
  ssh root@IP_СЕРВЕРА
  ```

- [ ] Обновить систему:
  ```bash
  apt update && apt upgrade -y
  ```

- [ ] Установить Docker и Docker Compose:
  ```bash
  curl -fsSL https://get.docker.com | sh
  ```

- [ ] Создать отдельного пользователя (не работать под root):
  ```bash
  adduser deploy
  usermod -aG docker,sudo deploy
  su - deploy
  ```

## 2. Загрузка кода

- [ ] Склонировать репозиторий:
  ```bash
  git clone <адрес-репозитория> shop
  cd shop
  ```

---

## 3. Настройка переменных окружения

- [ ] Создать `.env` на основе шаблона:
  ```bash
  cp .env.example .env
  ```

- [ ] Сгенерировать новый секретный ключ:
  ```bash
  python3 -c "import secrets; print(secrets.token_urlsafe(48))"
  ```

- [ ] Заполнить `.env` боевыми значениями:
  - `ENVIRONMENT=production`
  - `DEBUG=False`
  - `SECRET_KEY=` — сгенерированный выше ключ
  - `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` — новый надёжный пароль БД
  - `YOOKASSA_SHOP_ID`, `YOOKASSA_SECRET_KEY` — боевые ключи из ЛК YooKassa
  - `DATABASE_URL` и `REDIS_URL` — как в шаблоне (внутренние адреса Docker)

## 4. Запуск контейнеров

- [ ] Собрать и поднять production-конфигурацию:
  ```bash
  docker compose -f docker-compose.prod.yml up -d --build
  ```

- [ ] Проверить, что все контейнеры поднялись:
  ```bash
  docker compose -f docker-compose.prod.yml ps
  ```

## 5. Инициализация базы данных

Выполнять по порядку. Все команды — внутри контейнера `app`.

- [ ] Применить миграции:
  ```bash
  docker compose -f docker-compose.prod.yml exec app alembic upgrade head
  ```

- [ ] Создать стартовые категории (безопасный seed, не удаляет данные):
  ```bash
  docker compose -f docker-compose.prod.yml exec app python3 seed_prod.py
  ```

- [ ] Создать информационные страницы (доставка, возврат, размеры и т.д.):
  ```bash
  docker compose -f docker-compose.prod.yml exec app python3 seed_pages.py
  ```

- [ ] Создать администратора:
  ```bash
  docker compose -f docker-compose.prod.yml exec app python3 create_admin.py
  ```

> Тестовый `seed.py` на проде запускать НЕЛЬЗЯ — он удаляет данные
> и защищён предохранителем (`ENVIRONMENT=production`).
> Товары владелец добавляет через админку с реальными фото.

## 6. Nginx + HTTPS

Приложение слушает только `127.0.0.1:8000`. Наружу его отдаёт nginx с HTTPS.

- [ ] Установить nginx и certbot:
  ```bash
  sudo apt install -y nginx certbot python3-certbot-nginx
  ```

- [ ] Создать конфиг nginx для домена. В нём:
  - проксировать на `127.0.0.1:8000`
  - указать `client_max_body_size 10M` (для загрузки фото)
  - передавать заголовки `X-Forwarded-For` и `X-Forwarded-Proto`

- [ ] Получить SSL-сертификат:
  ```bash
  sudo certbot --nginx -d вашдомен.ru -d www.вашдомен.ru
  ```

- [ ] Проверить автопродление сертификата:
  ```bash
  sudo certbot renew --dry-run
  ```

## 8. Критичные доработки кода под прод

- [ ] **Чтение реального IP за nginx.** В роуте логина (`app/users/router.py`)
  для rate limiting используется `request.client.host`. За nginx все запросы
  придут с одного IP (адрес nginx). Нужно читать IP из заголовка
  `X-Forwarded-For`, иначе при срабатывании защиты заблокируются все
  пользователи разом.

- [ ] **Том для загруженных фото.** Папка `app/static/uploads/` должна
  переживать пересборку образа. Добавить именованный том для неё
  в `docker-compose.prod.yml`:
  ```yaml
  services:
    app:
      volumes:
        - uploads_data:/code/app/static/uploads
  volumes:
    uploads_data:
  ```
  (добавить том и для сервиса `app`, и в общий блок `volumes`).
  Без этого при `up --build` загруженные фото исчезнут.

- [ ] **Sentry (опционально).** Подключить отлов ошибок для мониторинга —
  узнавать о сбоях до того, как пожалуется покупатель.

## Приложение: вынос базы данных на отдельный сервер

В текущей конфигурации PostgreSQL работает в контейнере на том же сервере,
что и приложение. Для небольшого магазина этого достаточно.

Когда нагрузка вырастет, базу выносят на отдельный сервер (или управляемый
сервис — Managed PostgreSQL у облачных провайдеров). Зачем:
- база не конкурирует за ресурсы с приложением;
- провайдер берёт на себя бэкапы, обновления, отказоустойчивость;
- приложение можно масштабировать independently от базы.

Как это делается (общая схема, под конкретного провайдера уточняется):
1. Создать управляемую базу PostgreSQL у провайдера (Yandex Cloud, Selectel,
   Timeweb и др.) — получить host, port, пользователя и пароль.
2. Убрать сервис `db` из `docker-compose.prod.yml` (база теперь внешняя).
3. В `.env` прописать `DATABASE_URL` с адресом внешней базы вместо `@db:5432`.
4. Применить миграции к новой базе (`alembic upgrade head`).
5. Настроить доступ к базе только с IP сервера приложения (firewall).

Это шаг «на вырост» — для старта база в контейнере полностью рабочая.
